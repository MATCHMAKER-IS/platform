# ADR 0013: DB スキーマの適用は `db push`(マイグレーション履歴を持たない)

- 状態: 承認
- 日付: 2026-07-15

## 背景

Prisma には 2 つの適用方法がある。

| 方法 | 何をするか |
|---|---|
| `prisma db push` | schema.prisma と DB を**直接同期**する。履歴を残さない |
| `prisma migrate` | 差分の SQL ファイルを生成し、**履歴として積み上げる** |

現状:

- 開発(`scripts/setup.sh`)は **`db push`**
- **マイグレーションファイルは 1 つも無い**(`prisma/migrations/` が存在しない)
- しかし **CI(e2e.yml)は `prisma migrate deploy` を実行**していた → **マイグレーションが無いので何も適用されず、E2E が失敗する**

この不整合を解消するため、方針を決める。

## 決定

**`db push` を正式な方法とする。** マイグレーション履歴は持たない。CI もこれに揃える。

## 理由

- **本番はまだ稼働していない。** 「既存データを壊さずスキーマを変える」という、マイグレーションが本当に必要な場面がまだ来ていない。
- **db push は速く、迷いが無い。** 開発中は schema.prisma を書き換えて `db push` するだけ。マイグレーションを積むと、ブランチをまたぐたびに競合の解決が必要になる。
- **63 モデルを最初からマイグレーション化すると、初回の SQL が巨大になる。** レビューできない差分は、あっても意味がない。
- CI の E2E は**毎回まっさらな DB を作る**ため、履歴は不要(`db push` で最新スキーマにすれば足りる)。

## この決定の代償(理解した上で選ぶ)

| 失うもの | どうするか |
|---|---|
| スキーマ変更の履歴 | Git のコミット履歴(schema.prisma の差分)で追う |
| 本番での安全な差分適用 | **本番稼働を始める前に、この ADR を見直してマイグレーションへ移行する** |
| データ移行スクリプトの置き場 | 必要になったら `scripts/migrations/` を作る |

## 結果

- CI(e2e.yml)を `prisma migrate deploy` → **`prisma db push`** に変更する。
- SETUP.md の「チーム開発・本番はマイグレーション履歴を推奨」という記述を、この決定に合わせて改める。
- **本番稼働の前に必ずこの ADR を見直す**(下記の条件)。

## この決定を見直す条件(重要)

次のいずれかが起きたら、**マイグレーションへ移行する**。

- **本番運用を開始する**(消せないデータが入る)。これが最も重要な条件
- ステージング環境で「既存データを保ったままスキーマを変える」必要が出た
- 複数人が同時にスキーマを変更し、`db push` では取り込み順が問題になった

移行の手順:

```bash
# 1. 現在のスキーマを初期マイグレーションとして記録
pnpm --filter internal-app exec prisma migrate dev --name init

# 2. 以降はスキーマ変更のたびに
pnpm --filter internal-app exec prisma migrate dev --name add_task_table

# 3. 本番へは
pnpm --filter internal-app exec prisma migrate deploy
```
