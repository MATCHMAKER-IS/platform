# 0017: 権限は「付けたら終わり」にしない — 棚卸しと退職時の停止

- 日付: 2026-07-23 / 状態: 採用

## 文脈(なぜ決める必要があったか)

基盤には `@platform/auth` があり、「**今この人が何をできるか**」は正確に判定できる。
一方で、次のことを扱う仕組みも決めごとも無かった。

- **退職した人の権限が残っていないか**
- **異動した人に、前の部署の権限が残っていないか**
- 一時的に付けた権限(繁忙期の応援・障害対応)が、**そのままになっていないか**

権限は付けるときは慎重に検討されるが、**外すきっかけが無い**ため溜まり続ける。
実際、退職者のアカウントが半年後も生きていた、という事故はどの会社でも起きている。

内部統制(J-SOX)や ISMS でも定期的な確認が求められる項目であり、
「そのうちやる」では監査のたびに困る。

## 決定

**権限には必ず「見直す機会」を設ける。** 具体的には次の 3 つ。

### 1. 退職・異動のときに即座に止める(最優先)

| 事象 | やること | 期限 |
|---|---|---|
| 退職 | セッションの無効化 → ログイン不可 → 権限の削除 | **最終出社日のうち** |
| 異動 | 前部署の権限を外し、新部署の権限を付ける | 異動日のうち |
| 休職 | ログインを止める(権限は残す) | 休職開始日 |

**順序が大事。** 権限だけ消してもセッションが生きていれば操作できる。
セッションの無効化を先に行う。

人事の情報が Entra ID / Google Workspace にあるなら、**そこを起点にする**のが確実。
アプリごとに手で消すと必ずどこかが漏れる。

### 2. 半年に 1 回、全員の権限を棚卸しする

- **誰が・どの権限を・いつから持っているか**の一覧を出す
- 各部署の責任者が「必要」「不要」を判定する(情シスが判断しない。**業務を知る人が判断する**)
- 不要と判定されたものを外し、**判定した人と日付を記録する**

記録が無いと、監査で「確認しました」と言っても証明できない。

### 3. 強い権限には期限を付ける

`*`(全権限)や `pii:unmask`(個人情報の復号)のような強い権限は、
**恒久的に付けない**。付けるときは期限を決め、切れたら自動的に外れるようにする。

障害対応で一時的に管理者権限が要る場合も、**その場で付けて、終わったら外す**。
「いつか使うかもしれない」で持たせ続けない。

## 検討した代替案と見送り理由

| 案 | 見送り理由 |
|---|---|
| 権限の削除は退職手続きの一部として口頭で伝える | 忘れる。記録も残らない |
| 年 1 回の棚卸しにする | 半年前に辞めた人の権限が残る期間が長すぎる |
| すべての権限に期限を付ける | 通常業務の権限まで期限切れで止まると、業務が止まる。強い権限に絞る |
| 監査のときだけ棚卸しする | 監査の前だけ整うが、普段は溜まったまま。事故は監査の合間に起きる |

## 影響(良い点・受け入れるコスト・関連ドキュメント)

良い点:

- 退職者・異動者の権限が残る期間が短くなる
- 監査で「いつ・誰が確認したか」を示せる
- 強い権限が常時付いている人がいなくなる

受け入れるコスト:

- 半年に 1 回、各部署の責任者に確認を依頼する手間がかかる
- 期限付き権限は、切れたときに再申請が要る(**それが目的**でもある)

関連:

- `@platform/auth`(権限の判定)/ `packages/access-review`(棚卸しの支援)
