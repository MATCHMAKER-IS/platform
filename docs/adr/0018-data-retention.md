# 0018: 保存義務と削除要求が衝突したら、保存義務を優先し本人に説明する

- 日付: 2026-07-23 / 状態: 採用

## 文脈(なぜ決める必要があったか)

社内システムのデータには、**逆を向く 2 つの要求**がかかる。

| 要求 | 根拠 | 内容 |
|---|---|---|
| **消せ** | 個人情報保護法 | 利用目的の達成後は遅滞なく消去する。本人から削除を求められたら応じる |
| **残せ** | 電子帳簿保存法・法人税法 | 会計帳簿・証憑は 7 年間保存する |
| **残せ** | 労働基準法 | 労働者名簿・賃金台帳は 5 年間保存する |

基盤には両方の仕組みが既にあった(`@platform/pii` の `recordsToErase`、
`@platform/dencho` の `retentionDeadline`)。しかし**衝突したときにどちらを取るか**が
決まっておらず、`recordsToErase` の説明には「法令で保持が必要なものは除く」と書かれていたのに
**実装は除いていなかった**。

このままだと、削除要求に素直に応じたときに保存義務違反が起きる。

## 決定

**法令の保存義務が、削除要求に優先する。** ただし黙って残さず、**本人に説明する。**

### 判断の順序

1. **法令の保存義務があるか** → あれば残す。根拠と「いつ消せるか」を本人に伝える
2. 義務が無い(または切れた)なら、**自社の保持期間**で判断する
3. 保持期間も過ぎていれば消す

`decideErasure()` がこの順で判断し、`explainErasure()` が本人向けの説明文を作る。

### 本人への説明で必ず含めるもの

- **何件消したか / 何件残したか**
- **残した根拠**(「電子帳簿保存法 7 年」のように具体的に)
- **いつになれば消せるか**

「消せません」だけでは納得が得られない。根拠と期限を示す。

### 保存義務の記録

`LegalHold`(対象・根拠・期限)として持つ。**根拠を空にしない。**
空にすると、後から「なぜ残っているのか」を誰も説明できなくなる。

## 検討した代替案と見送り理由

| 案 | 見送り理由 |
|---|---|
| 削除要求には常に応じる | 保存義務違反になる。税務調査で帳簿が出せない |
| 保存義務があるものは黙って残す | 本人に説明していないため、個人情報保護法の観点で不十分 |
| 個人情報を含む部分だけ匿名化して残す | 会計帳簿は**取引先名を含めて**保存する必要がある。匿名化すると帳簿の要件を満たさない |
| 判断を都度、法務に確認する | 件数が多いと回らない。基準を決めて機械が判断し、例外だけ相談する |

## 影響(良い点・受け入れるコスト・関連ドキュメント)

良い点:

- 削除要求への対応が**機械的に判断できる**(担当者ごとに判断が揺れない)
- 本人への説明が定型化され、対応が早くなる
- 監査で「なぜ残しているか」を示せる

受け入れるコスト:

- どのデータに保存義務があるかを、**最初に整理する必要がある**(会計・人事・契約)
- 保存義務の期限が切れたものを消す仕組み(定期実行)が別途要る

関連:

- `@platform/pii` の `decideErasure` / `explainErasure`
- `@platform/dencho`(電子帳簿保存法の保存期間)
- `docs/ops/BACKUP_RESTORE.md`(バックアップにも同じデータが残る点に注意)
