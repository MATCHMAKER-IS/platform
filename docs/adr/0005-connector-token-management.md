# ADR 0005: 外部連携の OAuth トークン管理を基盤に取り込む

- ステータス: 承認
- 日付: 2026-07
- 関連: ADR 0002(基盤とアプリの分離)、`docs/platform/INTEGRATIONS.md`

## 背景

ADR 0002 の当初方針では、外部連携コネクタ(`@platform/zoho` `@platform/google`
`@platform/freee` 等)は「トークンを受け取って API を叩くだけ」とし、OAuth の認可から
アクセストークンのリフレッシュまでをすべてアプリ側の責務としていた。

しかし運用を進める中で、**トークンのリフレッシュは各サービスでほぼ定型**であることが分かった:
有効期限の管理、失効前バッファ、同時更新の1本化(inflight)、401 での再取得、
リフレッシュトークンのローテーション(freee)——これらを各アプリが個別に実装するのは
非効率で、実装漏れ(同時更新の競合や 401 ループ)も起きやすい。

## 決定

**OAuth トークンの自動更新を基盤側の責務に含める。** 各連携パッケージは、
アクセストークンを受け取るだけの薄いクライアントに加えて、リフレッシュトークンから
アクセストークンを自動更新する**トークンマネージャ**を提供する:

- `createZohoTokenManager` / `createGoogleTokenManager` / `createFreeeTokenManager`
- 共通の性質: 有効期限バッファ・同時更新の1本化・`onRefresh` による永続化フック・
  `createXxxAuthedFetch` による 401 自動リトライ。

一方、以下は引き続き**アプリの責務**とする(アプリ固有・秘匿性が高いため):

- OAuth の**初回認可フロー**(ログイン画面への誘導、認可コードの受領、初回トークン取得)。
- トークン・クライアントシークレットの**永続化と暗号化**(`@platform/crypto` `@platform/secrets`)。

## 帰結

- アプリはトークン管理の定型コードを書かずに済み、実装漏れによる障害が減る。
- 基盤は「薄く保つ」原則からは一歩踏み込むが、**ロジックではなく定型のインフラ処理**であり、
  副作用(`fetchImpl`・時刻)はすべて注入可能なため、基盤の設計原則(テスト可能・合成可能)は保たれる。
- 認可フローと秘密情報の保管はアプリに残るため、ADR 0002 の分離思想は維持される。
