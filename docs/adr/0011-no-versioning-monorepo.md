# ADR 0011: 基盤パッケージのバージョン管理はしない（モノレポ内製前提）

- 状態: 承認
- 日付: 2026-07-15

## 背景

`packages/` の 103 パッケージ（当時）はすべて <!-- doc-numbers:ignore --> `version: "0.1.0"` / `private: true` のまま据え置いている。一見すると「バージョン管理が機能していない」ように見えるため、意図を記録する。

アプリは `"@platform/xxx": "workspace:*"` で参照している。これは pnpm workspace の機能で、**npm レジストリを介さずローカルの `packages/` を直接見る**。つまり基盤を編集した瞬間からアプリはその新しいコードを使い、`npm install` も再ビルドも要らない。

## 決定

**基盤パッケージにセマンティックバージョニングを適用しない。** バージョンは `0.1.0` のまま固定し、`private: true` を維持する。外部公開もしない。

代わりに、バージョン番号が本来担う役割を次の仕組みで代替する。

| バージョンが担う役割 | このリポジトリでの代替 |
|---|---|
| 破壊的変更の通知 | `tools/api-surface.mjs` — 公開 export のスナップショット比較で削除/改名を検出。CI で落とす |
| 影響範囲の把握 | `pnpm platform:check` — 削除された API を **どのアプリ・デモが使っているか**まで示す |
| 変更履歴 | Git のコミット履歴 + `PLATFORM_SERVICES.md`（機能カタログ） |
| 依存の整合性 | `tools/check-deps.mjs`（循環・層破り）、`pnpm typecheck`（型の不整合） |

## 理由

- **単一リポジトリ・単一デプロイ単位**である。基盤とアプリが常に同時にデプロイされるため、「アプリ A は基盤 v1.2、アプリ B は v1.5」という状態が発生しない。バージョンを振っても、常に全員が同じものを使う。
- **バージョン番号は嘘をつきやすい**。「破壊的変更なのに minor を上げた」といった運用ミスは、レビューで見落とされる。スナップショット比較なら機械が確実に検出する。
- **属人化を防ぐ**という本リポジトリの方針上、「バージョンを上げるべきか」の判断を人に委ねるより、機械が検出する方が確実。
- 内製基盤であり、外部に配布しない。SemVer は「知らない相手に変更を伝える」ための規約であり、同じリポジトリを触る同僚に対しては過剰。

## 結果

- 基盤を変更したら `pnpm platform:check` で影響を確認し、`pnpm platform:sync` で生成物を更新する。
- 破壊的変更は CI（`api-surface`）が検出する。意図的なら `--update` でスナップショットを更新し、同じ PR で呼び出し側も直す。
- `.changeset/` は設定を残すが、変更セットは作らない（将来、外部配布する日が来たときの土台として残置）。

## 将来この決定を見直す条件

次のいずれかが起きたら再検討する。

- 基盤を**別リポジトリのアプリ**から使うようになった（＝レジストリ配布が必要になった）
- アプリごとに**異なるバージョンの基盤**を使う必要が出た（＝デプロイ単位が分かれた）
- 基盤を社外・グループ会社へ提供することになった
