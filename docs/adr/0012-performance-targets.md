# ADR 0012: パフォーマンスの目標値と測り方

- 状態: 承認
- 日付: 2026-07-15

## 背景

「システムが重い」という声は主観的で、対策の効果も測れない。一方、目標値を決めずに最適化すると、必要のない場所に時間を使う。**何を・いくつまで・どう測るか**を決めておく。

社内アプリの利用者は最大 200 名程度、同時アクセスのピークは朝の打刻時（9:00 前後の数分間）を想定している。

## 決定

### 目標値

**p95（95% のリクエストがこの時間以内）** を基準にする。平均値は外れ値に引きずられて実態を隠すため使わない。

| 対象 | p95 目標 | 根拠 |
|---|---|---|
| **ヘルスチェック** | 50ms | ここが遅いなら他は測るまでもない |
| **一覧の取得**（勤怠・経費・請求） | **300ms** | 人が「待った」と感じ始めるのが 300ms 前後 |
| **単純な書き込み**（打刻・申請） | **500ms** | 押した実感が得られる範囲 |
| **検索**（RAG・全文） | 1,000ms | 検索は待つものという期待がある |
| **重い集計**（月次決算・給与計算） | 30,000ms（max） | バッチ的な処理。p95 ではなく **max がタイムアウトしないこと**を見る |

### 絶対に守る線

| 項目 | 基準 |
|---|---|
| **エラー率** | **0%**（負荷時も）。1 件でも 5xx が出たら遅さの議論より先に原因を潰す |
| **朝の打刻（並列 200）** | エラー率 0% かつ p99 < 3,000ms |

### 測り方

```bash
pnpm dev:internal        # 対象を起動

# 別ターミナルで
pnpm loadtest -- --url http://localhost:3000/api/health --concurrency 10 --duration 10000
```

業務パターンでの測定は `demos/loadtest-scenarios` を使う（朝の打刻・経費ラッシュ・平常運転・月次決算）。

### 何を見るか

1. **エラー率**（0% でないなら、まずそれ）
2. **p95**（目標値と比較）
3. **ステップ別の p95**（全体が良くても特定 API だけ遅いことがある）
4. **max**（極端な外れ値は詰まりのサイン）

## 理由

- **p95 を選ぶ理由**: 平均は外れ値に弱く、p99 は少数の異常値に振り回される。「ほとんどの人の体感」を表すのは p95。
- **300ms の根拠**: 人間が「即座」と感じるのは 100ms 以内、「待った」と感じ始めるのが 300ms 前後、「遅い」と明確に認識するのが 1 秒（Nielsen の古典的な指標）。社内アプリなので 100ms は過剰、1 秒は遅すぎる。
- **エラー率 0% を絶対にする理由**: 打刻が 1 件でも失敗すると、その人の勤怠が欠ける。給与に直結するため妥協できない。
- **月次決算を p95 で測らない理由**: 実行回数が少なく（月 1 回）、パーセンタイルに意味がない。「タイムアウトせず完走するか」だけが問題。

## CI での扱い

**CI では性能を測らない。**

理由: GitHub Actions のランナーは性能が安定せず（実行ごとに 2〜3 倍ぶれる）、閾値を設けると**偽の失敗が頻発**して誰も見なくなる。「CI が赤いのはいつものこと」になるのが最悪。

代わりに:

- **リリース前に手元で測る**（`demos/loadtest-scenarios` の該当シナリオ）
- **本番では継続監視**（`@platform/observability` のメトリクス）
- **遅くなったという声が上がったら測る**（変更前後の比較）

## 測定の注意

- **開発機の絶対値は当てにしない。** 本番より遅い（Docker 経由の DB、開発ビルド）。**変更前後の比較**に使う
- **本番環境に負荷をかけない。** 本物の障害になる
- **1 回の測定を信じない。** 3 回測って中央値を見る

## 結果

- 目標値を超えたら、まず**どこが遅いか**を特定する（ステップ別 p95 → DB のクエリ → N+1 の有無）
- 目標値に届いていても、**時間とともに悪化していないか**を見る（メモリリーク・接続枯渇のサイン）
- この ADR の数値は**目安**であり、業務上の要求が変われば見直す

## 将来この決定を見直す条件

- 利用者が 1,000 名を超えた（同時アクセスの前提が変わる）
- 社外向けサービスを提供することになった（SLA が必要になる）
- CI ランナーが専用マシンになり、性能測定が安定するようになった
