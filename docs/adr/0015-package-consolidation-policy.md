# 0015: パッケージを分ける基準（統廃合の判断軸）

- 日付: 2026-07-22 / 状態: 採用

## 文脈(なぜ決める必要があったか)

`packages/` は 108 個・約 52,000 行まで育った。「多すぎるのでは」という懸念があったため実測したところ、
**使われていないパッケージは 0 個**だった。数そのものは問題ではなかった。

実際に問題だったのは重複である。例えば URL 関連では、

- `parseQuery` が `@platform/net` と `@platform/url` の**両方にあり、戻り値の型が違った**
- `joinUrl` が `net` と `blog` に**別実装で 2 つ**あった
- `buildQuery` と `stringifyQuery`、`withQuery` と `setParams` は**名前が違うだけの同じ機能**だった

この状態の害は「無駄」ではなく、**選べないこと**と**片側だけ古くなること**にある。
基準が無いまま増減を判断すると、その時々の気分で分かれてしまうため、基準を決める。

## 決定

パッケージを**分ける**のは、次のいずれかに当てはまるときだけとする。

1. **依存の重さが違う**
   Node 専用 API を使う / 重い外部ライブラリを持ち込む場合。混ぜるとブラウザ側のビルドが壊れる。
   例: `@platform/logger`(pino) は Node 専用。`@platform/net` は TCP/UDP を含むため、
   ブラウザから使う部分は `@platform/net/browser` として別入口にした。

2. **差し替える単位が違う**
   外部 SaaS ごと（freee / Google / Stripe / PayPal / Zoho …）。相手の都合で個別に壊れるため、
   分かれている方が影響範囲を閉じ込められる。

3. **上のどちらでもないなら、同じ関心事は 1 つにまとめる**
   「探したときに迷わない」ことを最優先する。**同じ機能が 2 か所にある状態を作らない。**

### 運用

- 新しい機能を足すときは、まず**既にある場所を探す**。無ければ既存パッケージに足す。
  新パッケージを作るのは 1 か 2 に当てはまるときだけ。
- 重複を見つけたら、**残す側を決めて片方を消す**。両方残して「どちらでもよい」にしない。
- 統廃合は **1 バッチ 1 テーマ**で行い、毎回 `pnpm check` と実ビルドを通す。
- 公開 API の削除は破壊的変更なので、意図した削除であれば
  `node tools/api-surface.mjs --update` で基準を更新し、その旨をコミットに書く。

### 名前の付け方

同じ機能に別名を付けない。迷ったら**動詞＋対象**で揃える（`setParams` / `parseQuery`）。
名前が違うだけの関数を増やすと、重複に気づけなくなる。

## 検討した代替案と見送り理由

| 案 | 見送り理由 |
|---|---|
| 108 → 30 程度へ大きく統合する | 実アプリで 64 個が動いている。壊す危険に対して得るものが少ない |
| デモでしか使っていない 26 個を消す | 「まだ使っていない」だけで、不要とは限らない。必要なときに作り直す方が高くつく |
| 数の上限を決める（例: 80 個まで） | 数は結果であって目的ではない。上限を守るための不自然な統合を招く |
| 現状維持（何も決めない） | 重複が増え続ける。実際に URL 周りで 3 種類の実装ができていた |

## 影響(良い点・受け入れるコスト・関連ドキュメント)

良い点:

- 「これはどこにありますか」に**一意に答えられる**
- 直す場所が 1 か所になり、片側だけ古くなる事故が無くなる
- 新パッケージを作る/作らないの判断で迷わない

受け入れるコスト:

- 統廃合のたびに import の書き換えと API 基準の更新が要る
- パッケージが大きくなる方向に働くため、中身の整理（ファイル分割）は別途必要
