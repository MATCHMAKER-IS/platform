# Cursor で開発する

**Cursor** は AI を組み込んだエディタです（VS Code をベースにしているので、見た目も操作もほぼ同じ）。このリポジトリは AI で開発しやすいように整えてあるので、Cursor と相性が良いです。

> VS Code でも開発できます。Cursor は「AI に手伝ってもらいたい人」向けの選択肢です。

> **このページの位置づけ**: 「**道具の使い方**」を書いています。
> 「**何を守るか**」（規約）は [CLAUDE.md](../../CLAUDE.md) にあり、**Cursor も自動で読みます**。
>
> | 知りたいこと | 見るファイル |
> |---|---|
> | Cursor の操作・設定・MCP 接続（このページ） | ↓ |
> | 守るべき規約（AI も人も共通） | [CLAUDE.md](../../CLAUDE.md) |
> | どう書くか（定型コード） | [../ai/patterns.md](../ai/patterns.md) |

## なぜこのリポジトリと相性が良いか

このリポジトリには、AI が正しく動くための下地が揃っています。

| 仕組み | AI にとっての意味 |
|---|---|
| `CLAUDE.md` | 「このリポジトリのルール」を AI が読める形で置いてある |
| `docs/ai/patterns.md` | 「どう書くのが正解か」の定型コード |
| **基盤カタログ MCP** | AI が自分で「その部品あるか？」を検索できる |
| `pnpm check` | AI の書いたコードが壊れていないか、1 コマンドで確認できる |

**AI に丸投げすると事故ります**が、この下地があると「既存部品を使わず似た関数を作る」といった典型的な失敗を減らせます。

## 1. 導入

### インストール

https://cursor.com からダウンロード（Windows / Mac 両対応・個人利用は無料枠あり）。

会社で使う場合は**ライセンスの確認**を忘れずに（Pro は有料）。

### VS Code の設定を引き継ぐ

初回起動時に「Import VS Code settings」が出ます。VS Code を使っていたなら押してください。拡張機能・キーバインド・テーマがそのまま移ります。

### このリポジトリを開く

```
ファイル → フォルダーを開く → clone した platform フォルダ
```

## 2. 最初にやる設定

### 推奨の設定（`.cursor/rules` は不要）

このリポジトリには `CLAUDE.md` があり、**Cursor も自動で読みます**（`.cursorrules` の代わりになります）。追加設定は不要です。

もし Cursor 専用の指示を足したい場合は、リポジトリ直下に `.cursorrules` を作れますが、**CLAUDE.md と二重管理になるので推奨しません**。

### 基盤カタログ MCP を繋ぐ（強く推奨）

AI が「その部品、基盤にありますか？」を自分で調べられるようになります。**これを繋ぐかどうかで、生成されるコードの質がかなり変わります。**

`Cursor Settings` → `MCP` → `Add new MCP server`:

```json
{
  "mcpServers": {
    "platform-catalog": {
      "command": "node",
      "args": ["--experimental-strip-types", "/絶対パス/platform/tools/mcp-catalog.mts"]
    }
  }
}
```

> `/絶対パス/` は実際のパスに置き換えてください（Mac 例: `/Users/yamada/dev/platform/...`、Windows 例: `C:\\Users\\yamada\\dev\\platform\\...`）

繋がると、AI が次を使えるようになります。

| ツール | 何ができる |
|---|---|
| `search_platform` | 「CSV 出力」で基盤を検索 |
| `find_examples` | 「請求書」で使用例（demos/）を検索 |
| `explain_rules` | 設計ルールを確認 |
| `describe_package` | 部品の README と API を読む |

詳細: [基盤カタログ MCP](../ai/mcp-catalog.md)

## 3. 基本の使い方

Cursor の AI 機能は 3 つです。**使い分けが大事**です。

| 機能 | ショートカット | 向いている作業 |
|---|---|---|
| **Tab（補完）** | 書いていると勝手に出る | 定型的な続きを書く |
| **Cmd+K（インライン編集）** | `Cmd+K` / `Ctrl+K` | 選択範囲を直す・小さな追加 |
| **Chat（対話）** | `Cmd+L` / `Ctrl+L` | 相談・調査・大きな変更 |

### Tab 補完

書き始めると灰色で続きが出ます。`Tab` で採用、`Esc` で拒否。

**注意**: 補完は「それっぽいコード」を出します。**基盤に既にある関数を無視して、似た関数を書き始めることがあります**。使う前に「これ、基盤にないか？」を疑ってください。

### Cmd+K（インライン編集）

コードを選択して `Cmd+K` → 「この関数にエラーハンドリングを追加」のように指示。

**得意**: リファクタ、型を付ける、コメントを書く、テストを足す
**不得意**: 「経費機能を作って」のような大きな話（Chat を使う）

### Chat（`Cmd+L`）

右側にチャット欄が開きます。**このリポジトリでは Chat がメイン**です。

## 4. このリポジトリでの開発の流れ

### 鉄則: 「まず基盤を探す」を AI にやらせる

このリポジトリの最大のルールは **「車輪の再発明をしない」** です。AI は放っておくと新しく書きます。

**良い聞き方**:

```
経費のCSV出力機能を作りたい。
まず search_platform で基盤に使える部品がないか探して。
無ければ作る方針を相談させて。
```

**悪い聞き方**:

```
CSVを出力する関数を書いて
```
→ 基盤に `@platform/csv` の `toCsv` があるのに、AI が新しく書いてしまう

### 実際の流れ

```
① Chat に相談する（Cmd+L）
   「経費一覧に CSV 出力を足したい。search_platform で部品を探して、
    find_examples で使用例も見せて」

② AI が基盤を検索 → @platform/csv の toCsv を提案

③ 実装させる
   「その方針で apps/internal-app/src/app/expenses/ に実装して」

④ 自分で読む ← 重要
   生成されたコードを必ず読む。分からない部分は「なぜこう書いたの？」と聞く

⑤ 検証する
   ターミナルで pnpm check

⑥ 直す
   エラーが出たら、エラー全文を Chat に貼る
```

### AI に渡すと精度が上がるもの

Chat では `@` でファイルや情報を渡せます。

| 書き方 | 渡すもの |
|---|---|
| `@ファイル名` | そのファイルの中身 |
| `@Codebase` | リポジトリ全体から関連箇所を探す |
| `@Web` | Web 検索（新しいライブラリの情報など） |
| `@Docs` | 登録したドキュメント |

**例**:

```
@docs/ai/patterns.md を参考に、@apps/internal-app/src/app/expenses/expenses-client.tsx
に CSV 出力ボタンを足して。色は CSS 変数を使うこと。
```

### 効く指示・効かない指示

| ✅ 効く | ❌ 効かない |
|---|---|
| 「search_platform で探してから実装して」 | 「いい感じにやって」 |
| 「@patterns.md の 9 番のやり方で」 | 「ベストプラクティスで」 |
| 「pnpm check が通るまで直して」 | 「バグがないようにして」 |
| 「エラー全文を貼る」 | 「エラーが出た」 |
| 「なぜこの実装にしたの？」 | （聞かずに信じる） |

## 5. デバッグ

### AI にエラーを渡すとき

**エラー全文をそのまま貼る**のが最も効きます。要約しないでください。

```
以下のエラーが出た。原因と直し方を教えて。

TypeError: Cannot read properties of undefined (reading 'map')
    at ExpenseList (expenses-client.tsx:42:18)
    ...
```

Cursor はターミナルのエラーも読めます。エラーが出た状態で `Cmd+L` → 「このエラーを直して」でも通じることがあります。

### 型エラーが出たとき

```
@apps/internal-app/src/server/expense.ts の型エラーを直して。
any を使わずに、正しい型を付けて。
```

> **`any` を使わせない**のは重要です。AI は困ると `any` で逃げます。型エラーは「設計がおかしい」サインのことが多いので、`any` で塞ぐと後で困ります。

### 動かないとき（画面が真っ白など）

AI に頼る前に、**自分で原因を絞ってください**。AI は状況を見ていません。

1. ブラウザの Console（F12）にエラーは出ているか
2. ターミナル（`pnpm dev` の画面）にエラーは出ているか
3. `pnpm doctor` — 環境の問題ではないか
4. `/admin/env` — 設定（API キー等）は入っているか

絞れたら、その情報ごと AI に渡します。

### ブレークポイントを使う

Cursor は VS Code と同じデバッガが使えます。

1. 行番号の左をクリック → 赤い点（ブレークポイント）
2. `F5` → 「Node.js」を選ぶ
3. 画面を操作すると、その行で止まる
4. 変数の中身を見る

「なぜこの値になるのか」は、AI に聞くよりデバッガで見た方が確実です。

## 6. テスト

### AI にテストを書かせる

```
@packages/theme/src/derive.ts の deriveTheme に対するテストを書いて。
@packages/color/src/index.test.ts を参考に、同じ形式で。
境界値（不正な色、極端な角丸）も含めて。
```

**参考ファイルを渡す**のがコツです。渡さないと、このリポジトリの流儀と違うテストを書きます。

### 検証は必ず自分で走らせる

```bash
pnpm check              # 型 + Lint + スモーク（これが基本）
pnpm test               # ユニットテスト
pnpm verify:offline     # 全ゲート（PR 前）
```

**AI に「テストは通りました」と言われても信じないでください。** 自分のターミナルで実行して確認します。

### AI が嘘をつく典型

| AI の主張 | 実際 |
|---|---|
| 「テストが通りました」 | 実行していない（実行できない場合がある） |
| 「この関数は基盤にあります」 | 存在しない名前をでっち上げた |
| 「型エラーを直しました」 | `any` で塞いだだけ |

対策: **`pnpm check` を自分で打つ**。これが唯一の真実です。

## 7. やってはいけないこと

| ❌ | なぜ |
|---|---|
| 生成コードを読まずにコミット | 何が入ったか分からない = ブラックボックス化。**このリポジトリが最も避けたいこと** |
| `packages/` を AI に大改造させる | 影響範囲が全アプリに及ぶ。基盤変更は慎重に |
| 「pnpm check が通ればヨシ」 | 通っても設計がおかしいことはある。読むこと |
| `.env` の中身を Chat に貼る | 秘密の値が外部に送られます |
| 大量のファイルを一度に変更させる | レビューできない = PR が通らない |

### 秘密情報の扱い

Cursor は入力内容をクラウドに送ります（Privacy Mode を除く）。

- **`.env` の中身、パスワード、API キーは貼らない**
- 会社のポリシーによっては `Privacy Mode` を有効にする（Settings → Privacy）

## 8. おすすめの使い方

### 得意なことに使う

| 用途 | 効果 |
|---|---|
| **調査**「経費まわりの実装どこ？」 | `@Codebase` で一発。grep より速い |
| **定型作業**「この形式でテストを 10 個」 | 手で書くより速く正確 |
| **説明**「このコード何してる？」 | 引き継ぎ・レビュー前の理解に |
| **リファクタ**「この関数を分割して」 | 機械的な作業は得意 |
| **エラー解決** エラー全文を貼る | 大抵は的確 |

### 苦手なことは任せない

| 用途 | なぜ苦手か |
|---|---|
| **設計判断**「アプリを分けるべき？」 | 業務背景を知らない。人が決める |
| **業務ルール**「経費の承認は何段階？」 | 会社ごとに違う。AI は知らない |
| **大きな新機能** | 途中で辻褄が合わなくなる。分割して頼む |

## 9. チェックリスト（AI が書いたコードをコミットする前）

- [ ] **自分で全部読んだ**（分からない箇所は聞いて理解した）
- [ ] 基盤に既存部品がないか確認した（`search_platform`）
- [ ] `pnpm check` を**自分で**実行して通った
- [ ] `any` や `@ts-ignore` で誤魔化していない
- [ ] 色をハードコードしていない（`var(--color-*)` を使っている）
- [ ] `process.env` を直接読んでいない（`server/env.ts` 経由）
- [ ] 消し忘れの `console.log` がない
- [ ] 基盤を変えたなら `pnpm platform:check` した

## トラブル

| 症状 | 対処 |
|---|---|
| MCP が繋がらない | パスが絶対パスか確認。Cursor を再起動 |
| AI が古い情報で答える | `pnpm gen:all` で生成物を更新 → MCP サーバを再起動（起動時読み込みのため） |
| 補完が邪魔 | `Cursor Settings` → `Tab` を無効化。Chat だけ使う |
| 動作が重い | 大きなファイルを開きすぎ。`@Codebase` は範囲を絞る |
| 存在しない関数を提案される | MCP を繋いでいないと起きやすい。繋いでください |

---

## まとめ

**Cursor は「速く書く道具」であって「考える道具」ではありません。**

- 設計は人が決める → AI に相談はしても、判断は自分
- 生成コードは必ず読む → 読まずに入れると、それがブラックボックスの始まり
- 検証は自分で打つ → `pnpm check` が唯一の真実

この 3 つを守れば、Cursor は非常に強力です。
