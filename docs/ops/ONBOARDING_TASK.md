# はじめての1機能(実地課題)

この基盤の目的は**属人化を防ぐこと**。しかし基盤そのものが一人しか触れない状態なら、
目的は達成できていない。それを確かめる唯一の方法は、**別の人に実際に作ってもらうこと**。

この文書は、その最初の課題。**読む人が作業し、教えた人が答え合わせをする。**

---

## この課題のねらい

- 新しく入った人が、**手順書だけで**画面を1つ作れるかを確かめる
- 詰まった箇所は「その人の理解不足」ではなく **基盤側の欠陥** として扱い、直す

> 詰まったら遠慮なく詰まってほしい。**詰まった場所こそが、この課題の成果物**。

## 課題

社内の**備品貸出**を記録する画面を1つ作る。

| 項目 | 内容 |
|---|---|
| 一覧 | 貸出中のものを表で表示する |
| 登録 | 「備品名」「借りた人」「貸出日」を入力して追加する |
| 返却 | 行ごとに「返却」ボタンを押すと一覧から消える |

**やらないこと**: 認証、DB、権限、デザインの作り込み。データは画面の状態(メモリ)でよい。

## 進め方

### 1. 動かす(目安 30 分)

```bash
pnpm install
pnpm doctor
pnpm dev:demos
```

- `http://localhost:3001` が開けたら成功
- **ここで詰まったら、それ以上進まずに記録する**(環境構築は最も詰まりやすい)

### 2. 参考にする画面を読む(目安 30 分)

このデモサイトで、近いものを探して**コードを読む**。

- `/master` … 一覧・登録・削除がそろっている。今回の課題に一番近い
- `/ui` … 使える部品の一覧
- `/faq` … 迷ったときの Q&A

該当ファイル: `demos/showcase/src/app/master/page.tsx`

### 3. 作る(目安 2〜3 時間)

`demos/showcase/src/app/lending/page.tsx` を新しく作る。

守ること(トップページの「開発の流れ」と同じ):

- `<button>` や `<input>` を直接書かず、`@platform/ui` の部品を使う
- `console.log` を使わない
- 失敗しうる処理は `Result` で返す(この課題では出番が無いかもしれない。無くてよい)

### 4. 確認する(目安 30 分)

```bash
node tools/preflight.mjs
```

全部緑になれば完成。赤くなったら、表示されたファイルと行を見て直す。

## 受け入れ基準(これが満たせたら合格)

- [ ] 一覧・登録・返却が画面上で動く
- [ ] 生の `<button>` / `<input>` を使っていない
- [ ] `node tools/preflight.mjs` が全部緑
- [ ] キーボードだけで登録から返却まで操作できる(Tab と Enter)
- [ ] **詰まった箇所を下に記録した**

## 教えた側がやること

課題が終わったら、**上の表を見て基盤を直す**。ここが本番。

| 記録された内容 | 直す場所 |
|---|---|
| 環境構築で詰まった | `docs/ops/GETTING_STARTED.md` / `pnpm doctor` の診断を増やす |
| どの部品を使えばいいか分からなかった | `/ui` デモ・トップページの「開発の流れ」 |
| 作法(Result / logger / env)の理由が分からなかった | `/faq` に Q&A を足す |
| エラーの意味が分からなかった | エラーメッセージ自体を直す(**メッセージは最も読まれるドキュメント**) |
| 手順書のとおりにやったのに動かなかった | 手順書を直す。`node tools/check-docs-links.mjs` で参照切れも確認 |

### 判定

- **手順書だけで完成した** … 基盤は他の人にも使える。次の人は別の課題へ
- **口頭の補足が必要だった** … その補足内容が、そのまま**不足しているドキュメント**
- **完成しなかった** … 基盤側の課題。作った人の問題ではない
