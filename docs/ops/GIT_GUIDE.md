# Git と GitHub の使い方

ソースコードの管理方法です。**Git を触ったことがない方**向けに、なぜ必要かから書いています。

すでに Git を使える方は [CONTRIBUTING.md](../../CONTRIBUTING.md)（このリポジトリ固有の約束）だけ読めば十分です。

---

## そもそも Git とは

**変更の履歴を記録し、いつでも戻せるようにする道具**です。

```
ファイルを直接いじる場合           Git を使う場合
─────────────────────           ─────────────────
app_v1.ts                        履歴が 1 本の線でつながる
app_v2.ts                        「いつ・誰が・なぜ」変えたか分かる
app_v2_最終.ts                    壊れたら任意の時点に戻せる
app_v2_最終_修正.ts  ← どれが最新？   複数人が同時に作業できる
```

**GitHub** は、その履歴を会社のサーバ（クラウド）に置いて共有する場所です。Git が「道具」、GitHub が「置き場」だと思ってください。

---

## 最低限おぼえる 5 つの言葉

| 言葉 | 意味 | 日常のたとえ |
|---|---|---|
| **リポジトリ** | プロジェクト 1 個分の入れ物 | プロジェクトのフォルダ |
| **コミット** | 変更の記録を 1 つ作ること | 「ここまでの作業を保存」 |
| **ブランチ** | 作業用の枝分かれ | 本番に影響しない作業用コピー |
| **プッシュ** | 自分の記録を GitHub に送る | クラウドにアップロード |
| **プルリクエスト（PR）** | 「この変更を取り込んで」と依頼する | 上司に承認をもらう |

---

## このリポジトリの進め方（全体像）

```
main ブランチ（本番。直接触れない）
  │
  ├─ feat/expense-report  ← 自分の作業ブランチ
  │    ↓ 作業してコミット
  │    ↓ GitHub にプッシュ
  │    ↓ プルリクエストを出す
  │    ↓ CI（自動テスト）が走る
  │    ↓ 同僚がレビュー
  └──← 承認されたら main に取り込まれる（マージ）
```

**`main` に直接 push することはできません**（保護されています）。必ずブランチを切って PR を出します。これは「間違って本番を壊す」のを防ぐためです。

---

## ツールを選ぶ

Git はコマンドでも GUI（画面）でも使えます。**最初は GUI がおすすめ**です。

| ツール | 費用 | 向いている人 | 対応 |
|---|---|---|---|
| **VS Code の Git 機能** | 無料 | **いちばんおすすめ**。エディタと一体で完結する | Win / Mac |
| **GitHub Desktop** | 無料 | GUI 専用で分かりやすい。VS Code を使わない人 | Win / Mac |
| **SourceTree** | 無料 | 履歴を figure で追いたい人。多機能だが重い | Win / Mac |
| **コマンドライン** | 無料 | 慣れたら最速。CI やトラブル対応で結局必要になる | Win / Mac |

**推奨**: VS Code の Git 機能で日常作業 → 困ったときだけコマンド。このページは両方書きます。

# 使い方: VS Code（推奨）

VS Code は最初から Git 機能が入っています。**追加インストール不要**です。

## 1. リポジトリを開く

`ファイル` → `フォルダーを開く` → clone したフォルダ（`platform`）を選ぶ。

左端のアイコン列に**枝分かれのマーク**（ソース管理）が出ます。これが Git の画面です。

## 2. ブランチを作る

画面**左下**に現在のブランチ名（`main`）が出ています。ここをクリック →「新しいブランチの作成」→ 名前を入力。

### ブランチ名の付け方

**形式**: `種類/内容-を-ハイフンでつなぐ`

| 種類 | いつ使う | 例 |
|---|---|---|
| `feat/` | 新機能を追加する | `feat/expense-attachment` |
| `fix/` | バグを直す | `fix/login-timeout` |
| `docs/` | ドキュメントだけ変える | `docs/setup-windows` |
| `refactor/` | 動作は変えずに整理する | `refactor/invoice-store` |
| `chore/` | 設定・依存の更新など雑務 | `chore/update-eslint` |
| `test/` | テストの追加・修正 | `test/payroll-edge-cases` |

**良い名前・悪い名前**:

| ❌ 悪い | 理由 | ✅ 良い |
|---|---|---|
| `test` | 何のテストか分からない | `test/payroll-rounding` |
| `修正` | 日本語は環境によって化ける | `fix/expense-validation` |
| `yamada-work` | 人名だと中身が分からない | `feat/monthly-report` |
| `feat/AddExpenseAttachmentFeatureToInternalApp` | 長すぎ・大文字は事故のもと | `feat/expense-attachment` |
| `feature/新機能` | 空白や日本語を混ぜない | `feat/expense-attachment` |

**ルール**:

- **小文字とハイフンだけ**使う（大文字・空白・日本語を避ける。OS によって扱いが変わり事故になります）
- **3〜5 単語**くらい。長すぎると `git checkout` が面倒
- **Issue 番号があれば入れてもいい**: `fix/123-login-timeout`

### ブランチはいつ作る・いつ消すか

```
作る: 作業を始めるとき（毎回。main で直接作業しない）
消す: PR がマージされたら（GitHub の画面に「Delete branch」ボタンが出ます）
```

**1 ブランチ = 1 目的**です。「経費機能を作るついでにボタンの色も直す」は分けてください。レビューが難しくなり、片方だけ戻したいときに困ります。

### 手元の古いブランチを掃除する

マージ済みのブランチが溜まってきたら:

```bash
# マージ済みのブランチ一覧を見る
git branch --merged main

# マージ済みを一括削除（main と現在のブランチは除く）
git branch --merged main | grep -v -E "main|\*" | xargs -r git branch -d

# GitHub 側で消えたブランチの参照を掃除
git fetch --prune
```

## 3. 変更を保存する（コミット）

1. ソース管理の画面を開くと、**変更したファイルが一覧**に出ます
2. ファイル名にマウスを乗せて **`+`** をクリック（＝この変更を記録対象にする＝ステージング）
3. 上のメッセージ欄に**何をしたか**を書く（例: `経費申請に添付機能を追加`）
4. **`✓ コミット`** ボタンを押す

> **コミットメッセージのコツ**: 「何をしたか」を日本語で。`修正` だけだと後で分かりません。
> 良い例: `経費申請の金額バリデーションを修正（0円を弾いていなかった）`

## 4. GitHub に送る（プッシュ）

コミットしたら、`変更の同期` または `ブランチの発行` ボタンを押します。初回は GitHub へのログインを求められます。

## 5. プルリクエストを出す

ブラウザで GitHub のリポジトリを開くと、**「Compare & pull request」の黄色いバナー**が出ています。それを押して:

- **タイトル**: 何をしたか（コミットメッセージと同じで OK）
- **説明**: なぜそうしたか、どう確認したか

「Create pull request」を押すと、**自動テスト（CI）が走ります**。緑のチェックが付けばレビュー依頼できます。

# 使い方: コマンドライン

VS Code で困ったとき、CI のログを読むとき、結局これが必要になります。

## 日常の 6 コマンド

```bash
# 1. 今どうなっているか見る（迷ったらこれ）
git status

# 2. 最新の main を取り込む（作業開始前に必ず）
git checkout main
git pull

# 3. 作業ブランチを作る
git checkout -b feat/expense-report

# 4. 変更を記録する
git add .                              # 全部の変更を対象にする
git commit -m "経費申請に添付機能を追加"

# 5. GitHub に送る
git push -u origin feat/expense-report   # 初回
git push                                  # 2回目以降

# 6. 履歴を見る
git log --oneline -10
```

## PR を出す（GitHub CLI を使う場合）

```bash
# 初回だけ: インストールとログイン
# Windows: winget install GitHub.cli
# Mac:     brew install gh
gh auth login

# PR を作る
gh pr create --title "経費申請に添付機能を追加" --body "確認: pnpm check 全緑"

# CI の結果を見る
gh pr checks
```

## 作業の流れ（毎回これ）

```bash
# ① 最新にする
git checkout main && git pull

# ② ブランチを切る
git checkout -b feat/my-feature

# ③ 開発する
pnpm dev

# ④ 壊れていないか確かめる（重要）
pnpm check

# ⑤ 記録して送る
git add .
git commit -m "何をしたか"
git push -u origin feat/my-feature

# ⑥ PR を出す（ブラウザ or gh pr create）
```

**④ を飛ばさないでください。** CI で落ちると気づくのが遅れ、レビュアーの時間も使います。

## このリポジトリ特有の注意

### 生成物を忘れずにコミットする

基盤（`packages/`）を変更したら、カタログや API 一覧を更新して**同じ PR に含めます**。

```bash
pnpm platform:check    # 影響を確認（削除した API を誰が使っているか）
pnpm platform:sync     # 生成物を更新
git add docs/          # 生成物もコミット対象
```

忘れると CI（`check-generated`）が落ちます。

### アプリと基盤を混ぜない

1 つの PR で「基盤に部品を追加」と「アプリでそれを使う」を両方やると、レビューが難しくなります。**基盤の PR を先に出して、マージされてからアプリの PR** を出してください。

### .env はコミットしない

`.env` には秘密の値（パスワード・API キー）が入ります。**`.gitignore` で除外済み**なので普通は大丈夫ですが、`git add .` の前に `git status` で確認する癖をつけてください。

## 競合（コンフリクト）が起きたら

**競合とは**: 自分と他の人が**同じファイルの同じ行**を変えたとき、Git が「どっちを採用すべきか分からない」と手を止める状態です。

**まず落ち着いてください。壊れていません。** Git が「人間に判断してほしい」と言っているだけです。

### いつ起きるか

```bash
git pull            # 最新を取り込もうとしたとき
# または PR の画面に「This branch has conflicts」と出たとき
```

### 画面ではこう見えます

```
<<<<<<< HEAD（現在の変更 = 自分）
const taxRate = 0.10;
=======
const taxRate = 0.08;
>>>>>>> main（入力側の変更 = 相手）
```

`<<<<<<<` から `>>>>>>>` までが競合箇所です。**この記号ごと消して、正しい状態にする**のがゴールです。

### VS Code / Cursor での直し方（推奨）

競合したファイルを開くと、ボタンが出ます。

| ボタン | 意味 |
|---|---|
| **現在の変更を維持** | 自分の変更を採用（相手のを捨てる） |
| **入力側の変更を維持** | 相手の変更を採用（自分のを捨てる） |
| **両方の変更を維持** | 両方残す（順番は自分で直す） |
| **変更の比較** | 並べて見る |

**どれを選ぶか迷ったら**: その行を変えた相手に聞いてください。**勝手に片方を消すのが一番危険**です（相手の修正が消える）。

### 手順（全体）

```bash
# 1. 最新の main を取り込む
git checkout main
git pull
git checkout feat/my-feature
git merge main              # ← ここで競合が出ることがある

# 2. 競合したファイルを直す（VS Code のボタン or 手で編集）
git status                  # 「both modified」が競合ファイル

# 3. 直したら「解決した」と伝える
git add .
git commit                  # メッセージは自動で入る

# 4. 送る
git push
```

### 競合を減らすコツ

- **こまめに `git pull`** する（1 日 1 回は main を取り込む）
- **PR を小さく・短命に**する（1 週間放置すると競合まみれになります）
- **同じファイルを同時にいじらない**ように、作業前に一声かける

### どうにもならなくなったら

```bash
git merge --abort     # マージをなかったことにする（競合前に戻る）
```

**最終手段**: 作業中のファイルを別フォルダにコピー → `git reset --hard origin/main` で綺麗にする → コピーを戻して作業し直す。**遠回りに見えて、これが一番速いこともあります**。

# プルリクエスト（PR）の使い方

## PR とは

**「この変更を main に入れていいか、見てください」という依頼**です。GitHub 上で、変更内容を見せながらレビューしてもらいます。

このリポジトリでは `main` が保護されているので、**PR 以外に本番へ反映する手段はありません**。

## 良い PR の作り方

### 1. 小さく出す

| ❌ | ✅ |
|---|---|
| 2 週間分・30 ファイルの変更 | 1〜2 日分・5 ファイル程度 |
| 「経費機能を作った」（機能全部入り） | 「経費申請の画面を追加」「経費の承認 API を追加」と分ける |

**大きい PR はレビューされません**（時間が取れないため後回しになる）。小さく出すと、その日のうちに見てもらえます。

### 2. タイトルと説明を書く

このリポジトリには**テンプレートが用意されています**（PR 作成時に自動で出ます）。埋めてください。

- **タイトル**: 何をしたか（`経費申請に添付機能を追加`）
- **確認したこと**: `pnpm check` を通したか、画面で動作を見たか
- **レビュアーに見てほしいところ**: 自信がない箇所を正直に書く

### 3. 自分で先にレビューする

PR を作ったら、**Files changed タブで自分の変更を全部見てください**。だいたい何か見つかります（消し忘れた `console.log`、コメントアウトしたコード等）。

## PR を出した後

```
① CI（自動テスト）が走る
   → 緑 ✅ になるまで待つ。赤 ❌ なら直す（Actions タブでログを見る）

② レビューを依頼する
   → 右側の「Reviewers」から選ぶ

③ 指摘が来たら直す
   → 同じブランチにコミットして push すれば、PR は自動で更新される

④ Approve をもらったらマージ
   → 「Squash and merge」を選ぶ（コミットを 1 つにまとめる）

⑤ ブランチを消す
   → 「Delete branch」ボタンが出る
```

## レビューで指摘されたときの心得

**コードへの指摘であって、人格への指摘ではありません。** 「なぜそうしたか」を説明すれば、相手が納得することもあります。議論は歓迎されます。

分からない指摘には「どうすればいいですか」と聞いて構いません。

# Issue（イシュー）の使い方

## Issue とは

**「やること」「困っていること」を記録する場所**です。バグ報告・機能要望・タスク管理に使います。

「口頭で言ったけど忘れられた」を防ぎます。**書いておけば消えません。**

## いつ書くか

| 場面 | 書く? |
|---|---|
| バグを見つけた（すぐ直せない） | **書く** |
| 「こうなったらいいのに」と思った | **書く** |
| すぐ直せる小さなバグ | 書かずに直して PR でいい |
| 使い方が分からない | まずドキュメント → それでも分からなければ人に聞く |

## 書き方

このリポジトリには**テンプレートが用意されています**（New issue を押すと選べます）。

### バグ報告で必ず書くこと

1. **何が起きたか** — 「どうなると思ったか」と「実際どうなったか」の**両方**
2. **再現手順** — 他の人が同じ状況を作れるように、順番に
3. **環境情報** — `pnpm doctor` の出力を貼る（環境が原因のことが多い）

> ❌ 悪い例: 「経費が動きません」
> ✅ 良い例: 「金額に 0 を入れて申請すると画面が固まる。期待はエラー表示。手順: 1. /expenses を開く 2. …」

### 機能要望で必ず書くこと

**「何が欲しいか」より「何に困っているか」**を書いてください。困りごとが分かれば、もっと良い解決策が見つかることがあります。

> ❌ 「CSV 出力ボタンが欲しい」
> ✅ 「毎月の経費集計で承認済みだけを手作業で Excel に貼っていて 30 分かかる。貼り間違いも起きる」

## Issue と PR をつなぐ

PR の説明に `Closes #123` と書くと、**マージ時に Issue が自動で閉じます**。

```
Closes #123     ← Issue 123 を閉じる
Refs #123       ← 関連はするが閉じない
```

# 用語集

Git / GitHub で出てくる言葉を、**実際に使う場面**とセットで。

## 基本

| 用語 | 読み | 意味 | 使う場面 |
|---|---|---|---|
| **リポジトリ** | repository | プロジェクト 1 個分の入れ物 | 「このリポジトリを clone して」 |
| **クローン** | clone | GitHub から手元にコピーする | 最初の 1 回 |
| **コミット** | commit | 変更の記録を 1 つ作る | 作業の区切りごと |
| **ブランチ** | branch | 作業用の枝分かれ | 作業を始めるとき |
| **プッシュ** | push | 手元の記録を GitHub に送る | コミットした後 |
| **プル** | pull | GitHub の最新を手元に取り込む | 作業を始める前 |
| **マージ** | merge | ブランチを合流させる | PR が承認された後 |

## PR まわり

| 用語 | 意味 |
|---|---|
| **プルリクエスト（PR）** | 「この変更を取り込んで」という依頼 |
| **レビュー** | 他の人が変更を確認すること |
| **Approve** | レビューで「OK」を出すこと |
| **コンフリクト（競合）** | 同じ行を別々に変えて、Git が判断できない状態 |
| **Squash and merge** | 複数のコミットを 1 つにまとめてマージ（このリポジトリの推奨） |
| **Draft PR** | 「まだ途中だけど見てほしい」PR |

## ちょっと難しい

| 用語 | 意味 | いつ出会うか |
|---|---|---|
| **ステージング** | コミットする変更を選んでおくこと | `git add` したとき |
| **HEAD** | 今いる場所（最新のコミット） | `git reset HEAD~1` などで |
| **origin** | GitHub 側のリポジトリの呼び名 | `git push origin main` |
| **リベース** | 履歴を作り直して繋げ直す | `git pull --rebase` |
| **fetch** | GitHub の情報だけ取ってくる（作業には反映しない） | `git fetch --prune` |
| **stash** | 作業を一時的に棚上げする | 急に別の作業が入ったとき |
| **cherry-pick** | 特定のコミットだけ持ってくる | 「あの修正だけ欲しい」とき |

### stash（作業の一時退避）

「作業途中だけど、急に別のブランチを見たい」ときに使います。

```bash
git stash              # 今の作業を棚に上げる（作業ディレクトリが綺麗になる）
git checkout main      # 別のブランチへ
# ... 用事を済ませる ...
git checkout feat/my-feature
git stash pop          # 棚から戻す
```

| 機能 | 何ができる |
|---|---|
| **`.` キー** | リポジトリの画面で `.` を押すと、ブラウザで VS Code が開く（軽い編集に便利） |

| **Draft PR** | 「作業中」と示して早めにフィードバックをもらう |

## よくある困りごと

| 症状 | 対処 |
|---|---|
| **間違ってコミットした** | `git reset --soft HEAD~1`（コミットだけ取り消し。変更は残る） |
| **コミットメッセージを間違えた** | `git commit --amend -m "正しいメッセージ"`（push 前のみ） |
| **変更を全部捨てたい** | `git checkout .`（**戻せません**。慎重に） |
| **main に直接コミットしてしまった** | `git branch feat/xxx`（今の状態でブランチを作る）→ `git reset --hard origin/main`（main を元に戻す）→ `git checkout feat/xxx` |
| **コンフリクト（競合）が出た** | VS Code が「現在の変更/入力側の変更」を選ばせてくれます。分からなければ**変更した人に聞く**のが速い |
| **push を拒否された** | 誰かが先に push している。`git pull --rebase` してから push |
| **何が起きているか分からない** | `git status` と `git log --oneline -5`。それでもダメなら**作業をコピーして clone し直す**のが確実 |

> **鉄則**: 分からなくなったら、**作業中のファイルを別フォルダにコピーしてから**操作してください。Git の操作ミスで失うより安全です。

## GitHub の画面の見方

| タブ | 何ができる |
|---|---|
| **Code** | ソースを見る・clone の URL を取得 |
| **Issues** | やること・バグの記録（「◯◯が動かない」を書く場所） |
| **Pull requests** | レビュー待ちの変更一覧 |
| **Actions** | 自動テスト（CI）の結果。**赤くなったらここを見る** |
| **Settings** | リポジトリの設定（管理者のみ） |

### CI が赤くなったら

1. **Actions** タブ → 赤い実行をクリック
2. 失敗したジョブ（赤い ✗）をクリック
3. ログの中の `❌` を探す

このリポジトリは失敗の要約を **PR に自動コメント**します（`ci-log-summary`）。まずそれを読むのが早いです。
