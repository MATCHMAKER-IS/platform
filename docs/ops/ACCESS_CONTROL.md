# 社内限定にする（アクセス制限）

社内向けに作ったものを、そのまま公開の場所に置くと**誰でも見られる**。
チャットボットのように**社内資料を答えるもの**は、特に注意が要る。

この文書は「誰に見せるか」を決めて、そのとおりに閉じるための手順。

## まず決めること

| 問い | 例 |
|---|---|
| 誰が見てよいか | 全社員 / 特定部署 / 管理者だけ |
| どこから見てよいか | 社内ネットワークのみ / VPN 経由 / どこからでも |
| 見えると困るものは何か | 資料の中身 / 誰が何を聞いたか（利用ログ）|

**「とりあえず社内だけ」で止めない。** 資料には給与や取引先の情報が混ざることがあり、
全社員に見せてよいとは限らない。

## 4 つの止め方（内側から順に）

### 1. ログインを必須にする（基本）

`middleware.ts` で、未ログインなら入口へ戻す。`apps/internal-app/src/middleware.ts` が実例。

```ts
// apps/<app>/src/middleware.ts の考え方
// 1. セッション Cookie を読む(@platform/guard の requireSession)
// 2. 無ければログイン画面へリダイレクト
// 3. 静的ファイルとログイン画面自身は除外する(matcher で指定)
```

`@platform/guard` が提供するもの:

| 関数 | 用途 |
|---|---|
| `requireSession` | ログインしているか。していなければ例外 |
| `requireRole` | 特定のロールを持つか |
| `requirePermission` | 特定の権限を持つか（RBAC。`/security` で挙動を確認できる） |

### 2. 権限で絞る（部署・役割）

全社員に見せてよいとは限らないので、ロールと権限で分ける。
判定は `@platform/auth` の `can()`。**「通ってはいけない」側**もスモークテストで固定してある
（存在しないロール・大文字違いで権限が通らないこと）。

```
例: 経理の資料 → finance ロールのみ
    人事の資料 → hr ロールのみ
    それ以外   → 全社員
```

チャットボットに適用する場合は、**検索の対象を利用者の権限で絞る**。
`@platform/rag` の `AccessControl` / `canAccess` が、チャンク単位の可視性を判定する。
回答を作る前に落とすので、**見えない資料が回答に混ざらない**。

### 3. 接続元を制限する（ネットワーク）

社内ネットワークや VPN からのみに限定する。IP の判定は `@platform/net/browser` の
`ipInCidr` / `isPrivateIp`（`/net` で挙動を確認できる）。

- 恒久的な制限は**インフラ側**（ロードバランサ・WAF・セキュリティグループ）で行う方が確実
- アプリ側の判定は補助。プロキシ経由だと `X-Forwarded-For` を見る必要があり、**偽装されうる**

### 4. そもそも公開しない

デモサイトのように「社外に見せる必要が無い」ものは、社内からしか届かない場所に置く。
公開する必要が出たときに 1〜3 を足す。

## チャットボット固有の注意

| 注意点 | なぜ |
|---|---|
| **索引に何が入っているか確認する** | `demos/showcase/public/docs-index.json` は資料の抜粋を含む。公開の場所に置けば**認証をかけても静的ファイルとして読める**ことがある |
| **利用ログの扱いを決める** | 「誰が何を聞いたか」は人事情報になりうる。残すなら期間と閲覧者を決める（`docs/ops/BACKUP_RESTORE.md` の保持期間の考え方と同じ） |
| **AI へ送る内容を絞る** | 資料に個人情報が混ざる場合、送信前にマスクする（`@platform/pii`。AI ゲートウェイの `redact` にも渡す） |
| **回答の根拠を残す** | どの資料を見て答えたかが分かれば、誤答時に原因を追える |

## 確認の仕方

閉じたつもりで閉じていない、が一番多い。**実際に試す。**

1. ログアウトした状態で URL を直接開く → 入口へ戻されるか
2. 権限の無いユーザーで開く → 見えないか
3. **静的ファイルを直接叩く**（例: `/docs-index.json`）→ 認証なしで読めていないか
4. 社外ネットワークから開く → 届かないか

3 は見落としやすい。middleware の `matcher` から静的ファイルを除外していると、
**画面は閉じているのにデータは読める**状態になる。

## 付けた権限を見直す

閉じ方を決めても、**権限は付けるだけだと溜まり続けます**。退職者のアカウントが半年後も生きていた、
という事故はどの会社でも起きます。方針は `docs/adr/0017-access-review.md` に定めました。

| いつ | やること |
|---|---|
| 退職・異動 | **最終出社日のうち**に停止（セッション無効化 → ログイン停止 → 権限削除の順） |
| 半年に 1 回 | 全員の権限を棚卸し。**業務を知る人**が要否を判定し、判定者と日付を記録する |
| 強い権限を付けるとき | 期限を決める（`*` / `pii:unmask` などは恒久的に付けない） |

`@platform/access-review` の `reviewAccess()` が、退職者・期限切れ・期限の無い強い権限・
長く見直していない権限を一覧にします。`offboardingSteps()` は退職時の手順を順序つきで返します。
