# バックアップと復元

> **バックアップは「取れていること」ではなく「戻せること」で完成する。**
> 取得だけして復元を試したことがない状態は、バックアップが無いのとほとんど同じ。

## 決めておく2つの数字

先に決めておかないと、障害の最中に「どこまで戻していいか」で止まる。

| 記号 | 意味 | 目標 | 意味するところ |
|---|---|---|---|
| **RPO** | どこまでのデータ損失を許容するか | **24 時間** | 最悪、前日のバックアップ時点まで巻き戻る |
| **RTO** | 何時間で復旧するか | **4 時間** | 検知から通常業務に戻るまで |

これはあくまで初期値。業務側と合意したら、この表を更新する。
「経理の締め日だけは RPO 1 時間」のように、業務ごとに変えてもよい。

## 何を守るのか

| 対象 | 失うと何が起きるか | 方式 |
|---|---|---|
| PostgreSQL のデータ | 業務データの消失。復旧不能なら業務停止 | 日次ダンプ + 世代保管 |
| アップロードされたファイル | 添付・帳票が開けなくなる | ストレージ側のバックアップ |
| `.env` / 秘密鍵 | 復元しても**復号できない**(暗号化データが読めない) | パスワード管理ツールで別途保管 |
| リポジトリ | 再デプロイ不能 | Git リモート(GitHub) |

**秘密鍵の保管漏れは典型的な失敗。** DB だけ戻せても、`@platform/crypto` の鍵が無ければ
暗号化した項目(マイナンバー等)は読めない。鍵は DB と**別の場所**に保管する。

## 取得する

```bash
# 日次ダンプ(圧縮・タイムスタンプ付き)
pg_dump "$DATABASE_URL" -Fc -f "backup-$(date +%Y%m%d).dump"
```

- **世代を持つ**(日次7世代・週次4世代など)。上書き1世代だけだと、
  「壊れたデータを含んだバックアップ」で上書きしてしまう事故に対処できない
- **別の場所に置く**。同じサーバの同じディスクに置いたバックアップは、
  ディスク故障で本体と一緒に消える
- 取得の**成功/失敗を通知する**(`@platform/notify`)。静かに失敗し続けるのが最悪

## 戻す

```bash
# 1. 空のデータベースを作る(既存に上書きしない。取り違え事故を防ぐ)
createdb restore_test

# 2. ダンプから復元する
pg_restore -d restore_test --no-owner --clean --if-exists backup-YYYYMMDD.dump

# 3. 中身を確認する(件数が想定どおりか)
psql restore_test -c "\dt"
psql restore_test -c "SELECT count(*) FROM \"User\";"
```

復元後は**アプリを実際に起動して**画面が出るところまで確認する。
テーブルが戻っただけでは「業務が再開できる」証明にならない。

## 復元訓練(半年に1回)

**訓練していない手順は、本番では動かない。** 半年に1回、実際に戻す。

手順:

1. 直近のバックアップを1つ選ぶ
2. 本番と**別の環境**に、上の「戻す」を最初から実行する
3. アプリを起動し、ログインして主要画面を1つ開く
4. かかった時間を測る(**RTO 4 時間に収まったか**)
5. 詰まった箇所をこの文書に反映する
6. `ops/drills/restore-drill.json` に記録する(下記)

記録:

```json
{
  "lastDrillAt": "2026-07-22",
  "durationMinutes": 95,
  "restoredFrom": "backup-20260721.dump",
  "operator": "(実施者)",
  "issues": ["秘密鍵の場所が手順に書かれておらず10分探した"],
  "nextActions": ["鍵の保管場所をこの文書に追記した"]
}
```

`node tools/check-drill.mjs` が訓練の間隔を見張り、**180 日を過ぎると警告**する。
preflight から呼ばれるので、放置すれば毎回目に入る。

## 削除したデータがバックアップに残る

本人からの削除要求に応じても、**バックアップの中には残ります**。世代保管をしていれば、
数か月前の断面に個人情報が入ったままです。

- バックアップの**保管期間を決める**（無期限に持たない）
- 復元したときは、**削除済みのデータを再度消す手順**を用意する
- 法令の保存義務があるものは、そもそも消さない（`docs/adr/0018-data-retention.md`）

## よくある失敗

| 失敗 | 対策 |
|---|---|
| バックアップは取れていたが、**復元したことが無い**ため手順で詰まる | 半年に1回の訓練 |
| 秘密鍵が無く、暗号化項目が読めない | 鍵を DB と別に保管し、訓練で**実際に復号まで確認する** |
| 世代が1つしかなく、壊れたデータで上書きされた | 世代保管 |
| バックアップが数か月前から失敗していた | 成功/失敗を通知する |
| 本番 DB へ間違えて復元した | 必ず**新しい空の DB** に戻す。本番への適用は別作業として承認を取る |
