# 障害対応（本番が止まったら）

**あわてないでください。** この順にやれば大丈夫です。

> **今すぐやること**: http://本番URL/admin/ops を開く → 赤い項目を見る → 下の該当箇所へ

---

## 0. 最初の 5 分

```
① 運用ダッシュボードを開く      /admin/ops
② 何が壊れているか特定           赤い ❌ を見る
③ 影響範囲を確認                 全員か・一部か・特定機能か
④ 関係者に一報                   「調査中」でいい。黙っているのが一番まずい
⑤ 下の「症状別の対応」へ
```

### 一報のテンプレート

```
【障害】社内アプリで◯◯ができない状態です
発生: 10:15 頃から
影響: 全員 / 経費申請のみ
状況: 調査中。分かり次第また連絡します
```

**原因が分からなくても連絡してください。** 「調査中」の一言があるかないかで、周囲の混乱が全く違います。

## 症状別の対応

### 画面が開かない（真っ白・エラー画面）

```bash
# 1. サーバは生きているか
curl https://本番URL/api/health
```

| 結果 | 意味 | 対応 |
|---|---|---|
| `{"ok":true}` | サーバは生きている | **画面側の問題**。ブラウザの Console を見る（[DevTools ガイド](DEVTOOLS_GUIDE.md)） |
| 応答なし・タイムアウト | サーバが落ちている | → 下の「サーバが落ちた」 |
| 502 / 503 | サーバは起動中か、落ちた直後 | 1 分待って再確認。それでもダメなら「サーバが落ちた」 |

### サーバが落ちた

```bash
# 1. プロセスは動いているか（デプロイ先で）
pm2 status              # ConoHa VPS の場合
# または
docker ps               # コンテナの場合

# 2. ログを見る（原因はほぼここに出ている）
pm2 logs --lines 100
# または
docker logs --tail 100 <コンテナ名>
```

**ログで探すもの**:

| ログの内容 | 原因 | 対応 |
|---|---|---|
| `環境変数の検証に失敗` | 設定漏れ | `.env` を確認。必須は `DATABASE_URL` / `SESSION_SECRET` / `MAIL_FROM` |
| `秘密値の強度が不十分` | 開発用の値のまま | 強いパスワードを生成して設定（[公開する](GETTING_STARTED_2.md#7-公開するデプロイ)） |
| `ECONNREFUSED` + ポート 5432 | DB に繋がらない | → 下の「DB に繋がらない」 |
| `JavaScript heap out of memory` | メモリ不足 | 再起動 → 根本対応は「メモリが足りない」へ |
| `EADDRINUSE` | ポートが使われている | 古いプロセスが残っている。`pm2 delete all` してから起動 |

```bash
# 3. とりあえず復旧させる
pm2 restart all
# または
docker compose restart
```

**再起動で直ったら、それで終わりにしないでください。** なぜ落ちたかをログから特定します（次に必ずまた落ちます）。

### DB に繋がらない

```bash
# 1. DB は生きているか
psql "$DATABASE_URL" -c "SELECT 1"
```

| 結果 | 対応 |
|---|---|
| `1` が返る | DB は生きている。アプリ側の設定（`DATABASE_URL`）を確認 |
| `Connection refused` | DB が落ちている。DB サービスを再起動 |
| `too many connections` | **接続が枯渇**。アプリを再起動すれば一旦回復。根本対応は接続プールの見直し |
| `password authentication failed` | 認証情報が違う。`DATABASE_URL` を確認 |

### 特定の機能だけ動かない

`/admin/ops` の該当項目を見ます。

| 項目が ❌ | 原因 | 対応 |
|---|---|---|
| **Zoho** | 外部 API の障害・認証切れ | サーキットブレーカーが開いています。**しばらく待つ**（自動で復旧を試みます）。長引くならトークンを確認 |
| **Webhook** | 配信先が落ちている | 相手側の問題。再送キューに溜まるので、復旧後に自動で流れます |
| **監査ログの整合性** | **改ざんの疑い** | **至急、責任者に報告**。`/admin/audit` で該当箇所を確認。触らずに保全 |

### メールが届かない

| 確認 | 方法 |
|---|---|
| 送信自体はできているか | ログに `mail sent` があるか |
| SMTP の設定 | `/admin/env` で `SMTP_HOST` / `MAIL_FROM` を確認 |
| 迷惑メール | 受信者の迷惑メールフォルダ |

> 開発環境では**実際には送信されず** Mailpit（http://localhost:8025）に届きます。本番と混同しないこと。

### 「遅い」と言われる

```bash
# 1. 実際に測る（体感は当てにならない）
pnpm loadtest -- --url https://本番URL/api/health --concurrency 5 --iterations 50
```

> **本番に負荷をかけすぎないでください。** 並列 5・50 回程度に留めます。

| 結果 | 対応 |
|---|---|
| p95 < 300ms | **遅くない**。特定の画面だけか確認（DevTools の Network で該当 API を見る） |
| p95 > 1000ms | 遅い。DB のクエリを疑う（N+1・インデックス漏れ） |
| エラーが混ざる | 遅さより先にエラーを潰す |

基準値: [ADR 0012](../adr/0012-performance-targets.md)

### 全員が使えない・原因不明

**メンテナンスモードにして、落ち着いて調べます。**

```
/admin/maintenance で ON
```

利用者には「メンテナンス中」の画面が出ます。中途半端に動いてデータが壊れるより安全です。

## 直し方が分からないとき

### ロールバック（前のバージョンに戻す）

**迷ったら戻す。** 原因究明は復旧してからで構いません。

```bash
# GitHub Actions からデプロイしている場合
# → Actions タブ → 前回の成功したデプロイ → Re-run jobs

# 手動デプロイの場合
git log --oneline -5          # 直前のコミットを確認
git revert <壊れたコミット>    # 打ち消しコミットを作る
git push                      # 再デプロイが走る
```

> **`git reset --hard` で歴史を消さないでください。** `revert`（打ち消し）を使います。何があったかの記録が残ります。

### DB のマイグレーションを戻す

**危険です。データが消えることがあります。** 必ずバックアップを取ってから。

```bash
# バックアップ（先にやる）
pg_dump "$DATABASE_URL" > backup-$(date +%Y%m%d-%H%M).sql

# 戻す
pnpm --filter internal-app exec prisma migrate resolve --rolled-back <マイグレーション名>
```

自信がなければ**やらないでください**。アプリを前のバージョンに戻すだけで済むことも多いです。

## 復旧したら

### 1. 関係者に連絡

```
【復旧】社内アプリの障害は 11:02 に復旧しました
原因: DB の接続数が上限に達していた
対応: アプリを再起動。恒久対策は別途検討します
```

### 2. 記録を残す（15 分でいい）

GitHub の Issue に書きます。**次に同じことが起きたとき、これが全てです。**

```markdown
## 何が起きたか
10:15〜11:02、経費申請が保存できなくなった。全員に影響。

## 原因
DB の接続数が上限（100）に達していた。月初の申請集中が引き金。

## どう直したか
アプリを再起動して接続を解放。

## 恒久対策
- [ ] 接続プールの上限を見直す
- [ ] 接続数のアラートを設定する

## 気づいたこと
/admin/ops を見れば 1 分で分かったが、最初 30 分ログを追っていた。
→ 障害時はまず /admin/ops、を周知する
```

### 3. 責めない

**人ではなく仕組みを直します。** 「誰がやったか」より「なぜ気づけなかったか」「どうすれば防げたか」です。

犯人探しをすると、次から誰も報告しなくなります。それが一番危険です。

## 予防（平常時にやること）

| いつ | やること |
|---|---|
| **毎日** | `/admin/ops` を眺める（30秒。異常の早期発見） |
| **リリース前** | `pnpm verify:offline` が全緑・負荷試験（[loadtest-scenarios](../../demos/loadtest-scenarios/README.md)） |
| **月初前** | 経費ラッシュを想定した負荷試験 |
| **月次** | バックアップから復元できるか試す（**試していないバックアップは無いのと同じ**） |

### バックアップ

```bash
# 取る
pg_dump "$DATABASE_URL" > backup-$(date +%Y%m%d).sql

# 戻す（テスト環境で試すこと）
psql "$DATABASE_URL" < backup-20260715.sql
```

**復元を試したことがないバックアップは、無いのと同じです。** 月に一度は試してください。
