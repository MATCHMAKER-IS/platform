# Chrome の開発ツール（DevTools）で調べる

ブラウザに標準で入っている調査ツールです。**「画面が動かない」の原因は、ほぼここで分かります。**

追加インストールは不要です。Chrome / Edge どちらも同じように使えます。

---

## 開き方

| 方法 | Windows | Mac |
|---|---|---|
| ショートカット | `F12` または `Ctrl+Shift+I` | `Cmd+Option+I` |
| 右クリック | 画面を右クリック →「検証」 | 同じ |
| メニュー | ⋮ → その他のツール → デベロッパー ツール | 同じ |

**特定の要素を調べたいとき**は、その要素を右クリック →「検証」。その要素が選択された状態で開きます。

### 邪魔なとき

右上の ⋮（DevTools 内）→ **Dock side** で、右・下・別ウィンドウを切り替えられます。画面が狭いなら別ウィンドウが快適です。

---

## タブの使い分け

| タブ | いつ見るか |
|---|---|
| **Console** | **まずここ。** エラーが出ていないか |
| **Network** | API が失敗していないか・遅くないか |
| **Elements** | HTML/CSS がどうなっているか（テーマの色を確認するときも） |
| **Application** | Cookie・localStorage（ログイン状態・テーマ選択） |
| **Sources** | ブレークポイントを置いて止める |
| **Performance** | 画面が重いとき |
| **Lighthouse** | 総合診断（性能・アクセシビリティ） |

---

# Console — まずここを見る

## 何が分かるか

赤いエラーが出ていれば、それが原因です。

```
❌ TypeError: Cannot read properties of undefined (reading 'map')
     at ExpenseList (expenses-client.tsx:42:18)
```

- **`Cannot read properties of undefined`** → データが来ていない（API 失敗を疑う → Network タブへ）
- **`is not a function`** → 関数名の間違い、または import 漏れ
- **`Hydration failed`** → サーバとブラウザで描画が違う（Next.js 特有。日付や乱数を直接描画していないか）
- **`Failed to fetch`** → API に繋がっていない（サーバが起動しているか）

**ファイル名と行番号（`expenses-client.tsx:42`）をクリック**すると、その場所に飛べます。

## 絞り込む

エラーが多すぎるときは、上部の絞り込みを使います。

- **Errors のみ表示**: 上の `Default levels` → Errors だけチェック
- **文字で絞る**: Filter 欄に `expense` など

## その場で試す

Console には**その場で JavaScript を実行**できます。

```js
// 今の画面の状態を見る
document.documentElement.dataset.skin      // 適用中のテーマ
localStorage.getItem("skin")               // 保存されているテーマ

// API を直接叩いてみる
await fetch("/api/expenses").then(r => r.json())
```

**これが便利**です。「API が悪いのか、画面が悪いのか」を切り分けられます。

> **注意**: 業務データを扱う画面では、Console に貼るコードに気をつけてください。他人のセッションで実行しないこと。

# Network — API を調べる

## 使い方

1. Network タブを開く
2. **画面をリロード**（開いてから操作しないと記録されません）
3. 一覧に通信が出る

## 見るポイント

| 列 | 意味 |
|---|---|
| **Status** | **200 なら成功。赤い数字（4xx/5xx）が問題** |
| **Name** | どの API か |
| **Time** | 何ミリ秒かかったか |
| **Size** | 応答の大きさ |

### ステータスの意味

| 番号 | 意味 | 対処 |
|---|---|---|
| **200** | 成功 | 正常 |
| **304** | 変更なし（キャッシュ） | 正常 |
| **400** | 送ったデータがおかしい | Payload タブで送信内容を確認 |
| **401 / 403** | ログインしていない / 権限がない | ログイン状態・ロールを確認 |
| **404** | その API が無い | URL のタイポ、ルートの配置ミス |
| **500** | サーバ側でエラー | **ターミナル（`pnpm dev` の画面）にスタックトレースが出ています** |
| **503** | サービス利用不可 | 設定が未構成のことが多い → `/admin/env` |

### 中身を見る

失敗した行をクリックすると、右側に詳細が出ます。

| サブタブ | 何が見える |
|---|---|
| **Headers** | URL・メソッド・ステータス |
| **Payload** | **送ったデータ**（POST の中身） |
| **Response** | **返ってきたデータ**（エラーメッセージが入っていることが多い） |
| **Preview** | Response を整形して表示（JSON が読みやすい） |
| **Timing** | どこで時間がかかったか |

**まず Response を見てください。** サーバがエラーの理由を返していることが多いです。

## 絞り込む

- **Fetch/XHR** ボタン: API だけ表示（画像や CSS を除外）
- **Filter 欄**: `expense` など、URL の一部で絞る

## 便利な機能

| 機能 | 使い道 |
|---|---|
| **Preserve log** | 画面遷移しても記録を残す（ログイン後のリダイレクトを追える） |
| **Disable cache** | キャッシュを無効化（DevTools を開いている間だけ有効） |
| **Throttling** → Slow 3G | **遅い回線を再現**。「外出先で使えるか」の確認に |
| 右クリック → **Copy as cURL** | ターミナルで同じリクエストを再現できる |

### 遅い回線を試す

Network タブ上部の `No throttling` → **Slow 3G** を選ぶと、遅い回線を再現できます。

「社内の Wi-Fi では快適だけど、出先で使い物にならない」を事前に見つけられます。

# Elements — 画面の構造を見る

## 何ができるか

- HTML の構造を見る
- **CSS をその場で書き換えて試す**（保存はされません）
- 要素の位置・大きさを確認

## このリポジトリで特に使うところ

### テーマ（CSS 変数）を確認する

このリポジトリの色はすべて CSS 変数です。適用されているか確認できます。

1. Elements タブで `<html>` を選ぶ
2. 右の **Styles** に `:root` や `[data-skin="..."]` の変数一覧が出る

```
data-skin="cute"        ← 適用中のスキン
data-theme="light"      ← ライト/ダーク

--color-primary: #ff7eb6
--color-bg: #fff5f8
--radius: 20px
```

**色がおかしいとき**は、ここで「変数が入っているか」「上書きされていないか」を見ます。

### その場で色を変えてみる

Styles の値をクリックして書き換えると、**即座に反映**されます（リロードで元に戻ります）。デザインの試行錯誤に便利です。

### 使われていない CSS を見つける

`Ctrl+Shift+P`（`Cmd+Shift+P`）→ `Coverage` → 記録 → リロード。使われていない CSS/JS が分かります。

# Application — 保存されているデータを見る

## Cookie

`Application` → `Storage` → `Cookies` → `http://localhost:3000`

| 名前 | 中身 |
|---|---|
| `session` | ログインセッション（封緘済み。中身は読めません） |

**ログインできない**ときは、ここに `session` があるか見ます。

## localStorage

`Application` → `Storage` → `Local Storage`

| キー | 中身 |
|---|---|
| `skin` | 選択中のテーマ（例: `cute`） |
| `theme` | ライト/ダークの設定 |

**テーマが元に戻ってしまう**ときは、ここが保存されているか確認します。

## 状態をリセットする

`Application` → `Storage` → **Clear site data**

「ログイン状態がおかしい」「テーマが変」なときに、まっさらな状態を試せます。

# Sources — ブレークポイントで止める

## 使い方

1. **Sources** タブ → 左のファイル一覧から対象を開く
   （見つからないときは `Ctrl+P` でファイル名検索）
2. **行番号をクリック** → 青い印（ブレークポイント）
3. 画面を操作すると、**その行で止まる**
4. 変数にマウスを乗せると中身が見える

## 止まったときの操作

| ボタン | 意味 | ショートカット |
|---|---|---|
| ▶ | 次のブレークポイントまで進む | `F8` |
| ⤵ | 次の行へ（関数には入らない） | `F10` |
| ↓ | 次の行へ（関数の中に入る） | `F11` |
| ↑ | 今の関数を抜ける | `Shift+F11` |

## 見るところ

| パネル | 内容 |
|---|---|
| **Scope** | 今のスコープの変数一覧（**これが一番使う**） |
| **Call Stack** | どこから呼ばれたか |
| **Watch** | 監視したい式を登録（`user.roles` など） |

## 条件付きブレークポイント

「100 回のうち特定の 1 回だけ止めたい」とき。

行番号を**右クリック** → `Add conditional breakpoint` → 条件を書く:

```js
expense.amount > 10000     // 1万円超のときだけ止まる
```

## console.log より速い

`console.log` を撒いて再起動するより、ブレークポイントの方が速く、消し忘れもありません。

**「なぜこの値になるのか」は Scope を見るのが最速**です。

# Performance — 画面が重いとき

## 使い方

1. **Performance** タブ → ● （記録）
2. 重い操作をする
3. ■ （停止）

## 見るポイント

| 色 | 意味 |
|---|---|
| **黄色（Scripting）** | JavaScript の実行。多いなら計算が重い |
| **紫（Rendering）** | レイアウト計算。多いなら DOM の操作が多すぎ |
| **緑（Painting）** | 描画 |

**赤い三角（Long task）** が出ていたら、そこが 50ms 以上ブロックしています。

## 簡易版: CPU を遅くして試す

Performance タブ → ⚙ → `CPU: 4x slowdown`

**非力な PC を再現**できます。「自分の PC では快適だけど、事務所の PC だと重い」を事前に見つけられます。

# Lighthouse — 総合診断

`Lighthouse` タブ → `Analyze page load`

性能・アクセシビリティ・SEO を点数で出してくれます。

## このリポジトリでの使いどころ

| 項目 | 見る意味 |
|---|---|
| **Performance** | 表示速度。公開サイト（:3004）では重要 |
| **Accessibility** | **色のコントラスト・ラベルの有無**。テーマを作ったら確認 |
| **Best Practices** | 一般的な問題 |
| **SEO** | 公開サイトのみ（社内アプリは検索避けなので無視） |

> テーマのコントラストは `pnpm smoke` でも検査していますが（`checkTheme`）、Lighthouse は**実際の画面**で見るので、組み合わせの問題を見つけられます。

# React の開発ツール（拡張機能）

React の状態を見るには、拡張機能を入れると便利です。

**インストール**: Chrome ウェブストアで「React Developer Tools」を検索

## 使い方

DevTools に **Components** と **Profiler** タブが増えます。

| タブ | 何ができる |
|---|---|
| **Components** | コンポーネントの階層・props・state を見る |
| **Profiler** | どのコンポーネントの再描画が重いか |

### このリポジトリで使うところ

**Components** で `SkinProvider` を選ぶと、今のテーマの状態が見られます。

```
SkinProvider
  props: { registry: {...}, mode: "light" }
  hooks:
    State: "cute"           ← 選択中のスキン
```

「テーマが切り替わらない」ときに、状態が更新されているかを確認できます。

# 症状から探す

| 症状 | まず見るところ |
|---|---|
| 画面が真っ白 | **Console** の赤いエラー |
| ボタンを押しても何も起きない | **Console**（エラー） → **Network**（API が呼ばれているか） |
| データが表示されない | **Network** → 該当 API の **Response** |
| ログインできない | **Application** → Cookie に `session` があるか |
| 403 と言われる | **Network** → Response（権限の詳細） → ログインユーザーのロール |
| 500 が返る | **ターミナル**（`pnpm dev` の画面）にスタックトレース |
| 色がおかしい | **Elements** → `<html>` の `data-skin` と CSS 変数 |
| テーマが元に戻る | **Application** → localStorage の `skin` |
| 画面が重い | **Performance** で記録 → 黄色（JS）が多いか |
| 「たまに」おかしい | **Network** の **Preserve log** を ON にして再現を待つ |
| メールが届かない | http://localhost:8025 （Mailpit。開発では実送信されません） |

# コツ

### 1. まず Console

9 割はここで分かります。**赤いエラーを読む**。読まずに推測しないこと。

### 2. 切り分ける

「API が悪いのか、画面が悪いのか」を Network で切り分けます。

- API が 200 でデータも正しい → **画面側の問題**
- API が 500 → **サーバ側の問題**（ターミナルを見る）
- API が呼ばれていない → **イベントが発火していない**（onClick の配線）

### 3. エラーは全文をコピーする

AI に聞くときも、人に聞くときも、**エラー全文**（スタックトレース込み）が最も情報量があります。要約しないでください。

### 4. Preserve log を使う

画面遷移すると Console/Network がクリアされます。ログイン → リダイレクトのような流れを追うときは、両方のタブで **Preserve log** を ON に。

### 5. 迷ったら Clear site data

「なんかおかしい」ときは、Application → Clear site data でまっさらにしてから再現を試します。**キャッシュや古いセッションが原因**のことが意外に多いです。
